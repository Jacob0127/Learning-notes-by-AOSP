# 一. 什么是Makefile

Makefile是`make`工具的配置文件，用于自动化代码编译、测试和部署过程。它定义了一系列**规则**来指定文件之间的依赖关系以及如何更新文件。

## 1.1 背景介绍

- 会不会写makefile，从侧面说明了一个人是否具备完成大型工程的能力
- 一个工程中的源文件不计其数，其按类型，功能，模块分别放在若干个目录中,makefile定义了一系列的规则来指定，哪些文件需要先编译，哪些文件需要后编译，哪些文件需要重新编译，甚至于更复杂的操作
- makefile 带来的好处就是 – “自动化编译”，一旦写好，只需要一个 make命令，整个工程完全自动编译，极大的提高了软件开发的效率。
- make 是一个命令工工具，是一个解释makefile中指令的命令工具，一般来说，大多数的IDE都有这个命令，比如：Delphi 的 make，Visual C++ 的 namke，Linux下GNU的make。可见，makefile都成为了一种在工厂方面的编译方法。
- make是一条命令，makefile是一个文件，两个搭配使用，完成项目自动化构建。

## 1.2 核心作用

- 定义源文件之间的**依赖关系**（如可执行文件依赖目标文件，目标文件依赖源文件）；
- 指定**编译规则**（如如何将.c文件编译为.o文件，如何链接为可执行文件）；
- 支持**增量编译**（只重新编译修改过的文件，减少重复工作）；
- 提供**项目清理功能**（一键删除编译产物，方便重新构建）。

# 二. 为什么使用Makefile

- **自动化构建**：一键执行复杂编译流程
- **增量编译**：只重新编译修改过的文件
- **跨平台支持**：兼容Unix/Linux/macOS/Windows
- **减少重复工作**：封装复杂命令
- **项目管理**：清晰定义项目结构

# 三.Makefile基本结构

## 3.1 基本规则语法

Makefile文件是由一系列规则（rules）构成。格式如下：

```shell
<target> : <prerequisites>
[tab] <commands>
```

- **target**：目标。**必需，不可省略**
- **prerequisites**：前置条件。**可选**
- **tab**：tab键。
- **command**：shell命令。**可选**
- 前置条件（prerequisite）和命令（command）**必须至少存在一个**

每条规则就是为了明确两件事：

1. 构建目标（target）的前置条件（prerequisite）是什么？
2. 如何构建（command）

### 目标

一个目标（target）就构成一条规则。通常是文件名，指明Make命令所要构建的对象。可以是一个文件名，也可以是多个，多个文件名中间用空格分隔。

除了文件名，目标还可以是某个操作的名字，这被称为“**伪目标**”（phony target）

```shell
clean:
	rm *.o
```

```bash
$ make clean
```

但是，如果当前目录中存在clean这个文件，那么这条命令不会被执行。

为了避免这种情况，可以明确声明clean是“**伪目标**”

```shell
. PHONY: clean
clean:
	rm *.o
```

> `.PHONY`：内置目标名
>
> 像这样的内置目标名很多，可以自己去了解

### 前置条件

前置条件通常是一组文件名，文件名之间用空格分隔。

它指定了“**目标**”是否需要重新构建的判断标准：**只要有一个前置文件不存在，或者有过更新（对比前置文件的时间戳和目标时间戳），“目标”就需要重新构建**

### 命令

**命令（commands）**表示如何更新目标文件，由一行或多行的Shell命令组成。

每行命令之前必须要有一个tab键。如果想用其他键，可以用**内置变量**`.RECIPEPREFIX`声明。

```shell
.RECIPEPREFIX = >
all:
> echo Hello, world
```

上面代码指定大于号（>)代替tab键。所以每一行命令的起首变成了大于号，而不是tab键。

需要注意的是，**每行命令在一个单独的shell中执行**。这些Shell之间没有继承关系。

```shell
test:
	export foo=bar
	echo "foo=[$$foo]"
```

上面代码执行`make test`，是取不到foo的值。因为两行代码在不同的进程执行。

一个解决办法是，将两行命令写在一行，中间用分号分隔。

```shell
test:
	export foo=bar; echo "foo=[$$foo]"
```

另一个方法就是，加上`.ONESHELL:`命令

```shell
.ONESHELL:
test:
	export foo=bar
	echo "foo=[$$foo]"
```

#### 回声

**首先，回声只会出现在命令中**

正常情况下，make会打印每条命令，然后再执行，这就叫回声（echoing）

```makefile
test:
	# 这是测试
```

执行上面的规则，会得到下面的结果

```bash
$ make test
# 这是测试
```

#### 关闭回声

在命令的前面加上@，就可以关闭回声

```makefile
test:
	@# 这是测试
```

现在再执行`make test`，就不会有任何输出。

> 但是由于在构建过程中，需要了解当前在执行哪条命令，所以通常只在注释和纯显示的echo命令前面加上@。

```makefile
test:
    @# 这是测试
    @echo TODO
```

## 3.2 简单示例

```makefile
# 编译C程序示例
app: main.o utils.o # 目标：app（最终可执行文件） 依赖：main.o utils.o（目标文件）
	gcc -o app main.o utils.o # 连接生成app

main.o: main.c # 目标：main.o 依赖：main.c（源文件）
	gcc -c main.c # 编译生成.o文件

utils.o: utils.c # 目标：utils.o 依赖：utils.c（源文件）
	gcc -c utils.c # 编译生成.o文件

# 删除所有.o文件和app
clean:
	rm -f *.o app
```

### 3.2.1 问题：既然前置条件是main.o和utils.o，那么这个app不应该不会生成吗，因为它的前置条件是在它之后生成的？

编译时的顺序是这样的：

1. 检查依赖，main.o和utils.o不存在
2. 先去执行前置条件的规则，也就是main.o和utils.o的生成规则
3. 有了之后，再编译成app
4. 最后删除所有.o文件和app

# 四. Makefile核心概念

## 4.1 定义变量

```makefile
CC = gcc
CFLAGS = -Wall -O2
TARGET = app
OBJS = main.o utils.o

test = ${CC}
```

## 4.2 自动变量

| 变量 | 含义                   |
| :--- | :--------------------- |
| `$@` | 当前规则的目标文件名   |
| `$<` | 第一个依赖文件名       |
| `$^` | 所有依赖文件列表       |
| `$?` | 比目标新的依赖文件列表 |
| `$*` | 匹配通配符的部分       |

```makefile
%.o: %.c
	$(CC) $(CFLAGS) -c $< -o $@
	
# 有文件：main.c utils.c
# 等价于：
main.o: main.c
    $(CC) $(CFLAGS) -c main.c -o main.o

utils.o: utils.c
    $(CC) $(CFLAGS) -c utils.c -o utils.o
```

## 4.3 使用变量

```makefile
$(TARGET): $(OBJS)
	$(CC) $(CFLAGS) -o $(TARGET) $(OBJS)

# 等价于
app: main.o utils.o
	gcc -Wall -O2 -o app main.o utils.o
```

## 4.4 变量的类别

### 4.4.1 递归扩展变量 =

在执行时扩展，允许递归扩展。这种变量，我们无法再对其采用赋值操作。

```makefile
.PHONY: all

foo=$(foo2)
foo2=$(foo3)
foo3=FLY.
	
all:
	@echo $(foo)
	
# 输出结果
FLY.
```

### 4.4.2 简单扩展变量 :=

在定义时扩展。这种变量，make只对其进行一次扫描和替换。

```makefile
.PHONY:all

x=fly
y=$(x) FLY
x=later
xx:=fly
yy:=$(xx) FLY
xx:=fly

all:
	@echo "x=$(y),xx=$(yy)"
	
# 输出结果
x=later FLY,xx=fly FLY
```

### 4.4.3 条件赋值变量 ?=

仅当变量未定义（或为空）时才赋值。

```makefile
.PHONY:all

foo=x
foo?=y
bar?=y

all:
	@echo "foo=$(foo),bar=$(bar)"
	
# 输出结果
foo=x,bar=y
```

### 4.4.4 追加赋值变量 +=

将值追加到变量的尾端。

```makefile
.PHONY: all

objects = main.o foo.o bar.o utils.o
objects += another.o

all:
	@echo $(objects)
	
# 等价于
.PHONY:all

objects = main.o foo.o bar.o utils.o
objects:= $(objects) another.o

all:
	@echo $(objects)

# 输出结果
main.o foo.o bar.o utils.o another.o
```

## 4.5 静态扩展和动态扩展

静态扩展指的是，**变量定义时立刻展开，结果固定不变**。而动态扩展是指**，变量使用时展开，结果自动更新**。

**声明变量时，默认都是静态扩展的，如果需要使用动态扩展，需要特殊写法。**

```makefile
a=1
b=${a} # 等于号（=）默认是静态扩展

b=${shell ...} # 动态扩展，最常用动态方式是<${shell}>
```

## 4.6 伪目标

一个目标（target）就构成一条规则。通常是文件名，指明Make命令所要构建的对象。可以是一个文件名，也可以是多个，多个文件名中间用空格分隔。

除了文件名，目标还可以是某个操作的名字，这被称为“**伪目标**”（phony target）

```makefile
clean:
	rm *.o
```

```bash
$ make clean
```

但是，如果当前目录中存在clean这个文件，那么这条命令不会被执行。

为了避免这种情况，可以使用`.PHONY:`来声明clean是“**伪目标**”

```makefile
.PHONY: clean
clean:
	rm *.o
```

## 4.7 函数

### 4.7.1 自动获取文件列表

```makefile
# 格式
$(wildcard pattern...)

# 例子
SRCS = $(wildcard src/*.c)  # 当前目录下所有.c文件
```

### 4.7.2 批量替换文件扩展名

```makefile
# 格式
$(patsubst pattern,replacement,text)

# 例子
OBJS = $(patsubst %.c,%.o,$(SRCS))  # .c → .o
```

### 4.7.3 批量添加路径前缀

```makefile
# 格式
$(addprefix prefix,text)

# 例子
INCLUDES = $(addprefix -I,include1 include2)  # -Iinclude1 -Iinclude2
```

## 4.8 条件判断

```makefile
DEBUG = 1

ifeq ($(DEBUG),1) # 如果DEBUG等于1
    CFLAGS += -g
else
    CFLAGS += -O2
endif
```

## 4.9 循环

```makefile
LIST = one two three
all:
    for i in $(LIST); do \
        echo $$i; \
    done

# 等同于
all:
    for i in one two three; do \
        echo $i; \
    done
    
# 运行结果
one
two
three
```

# 五. 完整项目示例

```makefile
# Makefile for C project

# 编译器配置
CC = gcc
CFLAGS = -Wall -Wextra
LDFLAGS = -lm

# 调试模式配置
DEBUG ?= 0
ifeq ($(DEBUG),1)
    CFLAGS += -g -O0
else
    CFLAGS += -O2
endif

# 目录配置
SRC_DIR = src
BUILD_DIR = build
BIN_DIR = bin

# 文件发现
SRCS = $(wildcard $(SRC_DIR)/*.c)
OBJS = $(patsubst $(SRC_DIR)/%.c,$(BUILD_DIR)/%.o,$(SRCS))
TARGET = $(BIN_DIR)/app

# 主构建规则
$(TARGET): $(OBJS)
	@mkdir -p $(@D)
	$(CC) $^ -o $@ $(LDFLAGS)

# 编译单个源文件
$(BUILD_DIR)/%.o: $(SRC_DIR)/%.c
	@mkdir -p $(@D)
	$(CC) $(CFLAGS) -c $< -o $@

# 伪目标
.PHONY: all clean run debug

all: $(TARGET)

clean:
	rm -rf $(BUILD_DIR) $(BIN_DIR)

run: all
	@echo "Running application..."
	@$(TARGET)

debug:
	@$(MAKE) DEBUG=1
```

# 六. Makefile最佳实践

1. **使用变量**：避免硬编码路径和参数
2. **目录分离**：区分源码、构建和二进制目录
3. **伪目标声明**：标记不生成文件的目标
4. **错误处理**：在命令前添加`-`忽略错误
5. **清晰的注释**：解释复杂规则和变量
6. **依赖管理**：自动生成头文件依赖
7. **分层Makefile**：大型项目使用子目录Makefile
8. **跨平台支持**：处理不同OS的差异
